# 数字货币全自动跟单系统产品逻辑文档

**作者：** Manus AI  
**更新时间：** 2026-03-31

---

## 1. 产品定位与核心价值

本系统是一个面向加密货币衍生品交易的全自动跟单平台。其核心价值在于将专业交易员（信号源）的交易策略实时、精准地复制到普通用户的独立交易所账户中，同时通过多级分销和自动化利润分成机制，实现平台、交易员、推广者和用户的多方共赢。

系统支持**跨交易所跟单**，信号源可以在 OKX 交易，而用户可以在 Binance、OKX、Bybit、Bitget、Gate.io 等多个主流交易所自动执行对应比例的订单。

---

## 2. 核心业务流程

### 2.1 账户与分销体系

本系统采用强绑定的邀请机制，所有新用户注册必须填写邀请码。这构成了平台的分销和利润分成基础。

- **注册与绑定：** 用户通过邮箱注册，必须输入上级的邀请码。系统自动生成该用户的专属邀请码，并建立 `referrerId` 推荐关系树。
- **层级关系：** 用户的层级关系决定了后续盈利订单的利润分成流向。系统支持无限级或多级差额分成计算。
- **交易所 API 绑定：** 用户在个人中心绑定自己真实交易所的 API Key 和 Secret（部分交易所需要 Passphrase）。系统对这些凭证进行强加密存储（AES-256-GCM），并在绑定时进行连通性测试。

### 2.2 信号源与策略管理

信号源是交易信号的发出者，由管理员在后台配置和管理。

- **信号源接入：** 管理员配置信号源的名称、交易对（如 ETH-USDT-SWAP）、参考仓位、预期月化收益等。
- **信号获取方式：**
  - **WebSocket 监听：** 核心引擎 (`copy-engine.ts`) 通过 OKX 的私有 WebSocket 频道，实时监听信号源账户的仓位变化（开仓、平仓、加仓、减仓）。
  - **Webhook 接收：** 系统也提供标准的 Webhook 接口，允许外部量化系统（如 TradingView 或 Python 量化引擎）主动推送交易信号。
- **用户订阅：** 用户在前端“策略中心”浏览可用的信号源，选择绑定好的交易所 API，设置**跟单倍数（Multiplier）**，并开启策略。开启策略前，系统会校验用户平台余额是否充足。

### 2.3 核心跟单执行引擎 (Copy Engine)

跟单引擎是系统的心脏，负责毫秒级的信号处理和订单分发。

- **信号解析：** 当监听到信号源仓位变化时，引擎计算出变化方向（做多/做空）和变化数量（合约张数）。
- **统一单位换算：** 由于不同交易所的合约规格不同（如 OKX 按张数，Binance 按基础币种数量），引擎统一将信号转换为基础资产数量（如 ETH 数量）：
  > `基础数量 = 信号源变动张数 × 信号源合约面值(ctVal) × 用户设置的倍数`
- **并发下单：** 引擎查出所有订阅了该信号且状态正常的用户，根据各自选择的交易所和计算出的最终数量，并发调用对应交易所的 API 执行下单操作。
- **容错与超时：** 引擎设有信号时效性校验（如 30 秒内有效），过期信号将被丢弃以防止极端行情下的滑点风险。下单失败会记录错误日志并在前端展示。

### 2.4 订单状态与盈亏结算

系统详细记录每一笔跟单操作的生命周期。

- **开仓单：** 引擎成功开仓后，生成状态为 `open`（持仓中）的订单记录。
- **平仓单：** 当信号源平仓时，引擎执行平仓操作，并将状态标记为 `closed`（已平仓）。
- **盈亏获取：** 平仓完成后，引擎会调用交易所的订单详情 API，获取该笔平仓单的**真实成交均价、手续费和已实现盈亏 (Realized PnL)**。系统不自行计算盈亏，完全以交易所结算数据为准，确保数据绝对准确。

### 2.5 资金与多级利润分成 (Revenue Share)

系统不触碰用户的交易本金（资金在用户自己的交易所账户中），但平台会收取跟单服务费，这需要用户在平台预存 USDT 作为余额。

- **预存资金：** 用户通过系统分配的专属 BSC 链地址充值 USDT。系统底层集成 HD 钱包，自动扫描链上交易并为用户入账。
- **差额分成逻辑：** 
  当用户的一笔跟单产生**净利润**（已实现盈亏 - 手续费 > 0）时，系统触发分成机制：
  1. 系统按用户设定的分成比例（如 50%）从用户的平台余额中扣除对应金额。
  2. 扣除的金额按**差额机制**向上级分配。例如：用户比例 50%，直接上级 A 比例 30%，上上级 B 比例 10%。
     - A 赚取：利润 × (50% - 30%) = 20%
     - B 赚取：利润 × (30% - 10%) = 20%
     - 剩余 10% 归平台所有。
  3. 分成实时结算到各级推荐人的平台余额中，推荐人可随时申请提现。
- **余额不足熔断：** 如果用户平台余额扣至 0，系统将自动暂停该用户的所有跟单策略，直到其重新充值。

### 2.6 积分与止损补偿系统

为了安抚产生亏损的用户，系统设计了积分补偿机制。

- **亏损兑换积分：** 用户在跟单中产生的净亏损，可以在每月指定时间兑换为平台积分。
- **积分流转：** 积分不仅可以作为平台活跃度的证明，还支持用户之间通过邀请码进行转账流通。

## 3. 技术架构概述

本系统采用现代化的全栈 TypeScript 架构，确保前后端类型安全和高效开发。

### 3.1 技术栈

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| **前端** | React, Vite, TailwindCSS | 采用 SPA 架构，界面响应迅速，组件化程度高 |
| **后端** | Node.js, tRPC, Fastify | tRPC 提供端到端的类型安全，无需手写 API 文档 |
| **数据库** | MySQL, Drizzle ORM | 关系型数据库保证交易和资金数据的一致性 |
| **部署** | PM2, Nginx | PM2 Cluster 模式充分利用多核 CPU，Nginx 作为反向代理 |

### 3.2 关键模块与文件分布

- **`server/copy-engine.ts`**：核心跟单引擎，负责 WebSocket 监听、单位换算、并发下单和盈亏回写。
- **`server/revenue-share.ts`**：多级差额利润分成算法的实现。
- **`server/bsc-wallet.ts`**：基于 ethers.js 的 HD 钱包实现，负责生成充值地址和扫描链上 USDT 充值。
- **`server/routers/`**：tRPC 路由层，包含 `strategy.ts` (策略与订单)、`funds.ts` (资金与充提)、`auth.ts` (认证与注册) 等核心业务接口。
- **`drizzle/schema.ts`**：数据库表结构定义，包含用户、订单、资金流水、分成记录等 12 张核心表。

## 4. 总结

该系统通过精巧的单位换算逻辑解决了跨交易所跟单的难题，利用多级差额分成模型构建了强大的自传播裂变能力，并通过预存服务费的方式规避了直接托管用户资金的合规风险。整体架构清晰，核心逻辑严密，具备支持数百人并发跟单的商业化运营能力。
